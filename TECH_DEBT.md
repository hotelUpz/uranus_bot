2. Внедрение Grid Take-Profit (Сеточная модель)
    1. Цель
    Заменить динамический поиск цели (hunting из BaseScenario) на статичную сетку лимитных ордеров (Grid TP). Параметры сетки (отступы в % и доли объема в %) зависят от метрики актива, определяемой через карту PHEMEX_TP_MAP.

    2. Маппинг и определение метрики
    Ключи карты (0, 500) и т.д. — это диапазоны некой метрики. Исходя из существующего кода (в PriceCacheManager есть get_volume), логичнее всего использовать суточный объем торгов (24h volume in USD) монеты.
    Чем выше объем (ликвидность), тем уже могут быть тейки. Чем ниже объем (щиткоины), тем шире раскидываем сетку.

    3. Изменения в Моделях Состояния (CORE/models_fsm.py)
    Чтобы Оркестратор понимал, что сетка выставлена и контролировал ее, в ActivePosition нужно добавить новые поля:

    Python
    # Флаг того, что сетка успешно рассчитана и отправлена на биржу
    tp_grid_initiated: bool = False

    # Словарь ID выставленных лимиток -> информация об уровне (для трекинга и аналитики)
    # Формат: { "order_id_123": {"idx": 1, "price": 105.5, "qty": 10.0} }
    tp_orders: dict = field(default_factory=dict)

    # Прогресс (сколько тейков уже исполнилось). Полезно для будущего переноса стопа в БУ.
    tp_progress: int = 0
    4. Новый модуль чистой математики (вместо EXIT/scenarios/base.py)
    Создаем класс (например, GridTPFactory), который занимается только расчетами. Сетевых вызовов внутри нет.

    Входящие данные метода расчета:

    symbol, side (LONG/SHORT)

    entry_price, current_qty (из ActivePosition)

    Спецификации монеты: price_precision (tick_size), lot_size

    Текущий объем торгов монеты (volume_24h_usd)

    Логика работы:

    По volume_24h_usd находим нужный бакет в PHEMEX_TP_MAP.

    В цикле проходимся по массивам levels и volumes.

    Считаем target_price = entry_price * (1 ± level/100), округляем по tick_size.

    Считаем cur_qty = total_qty * (volume/100), округляем по lot_size.

    Важный нюанс: На последнем шаге (последний тейк) нужно отдать весь оставшийся объем, чтобы не оставить на балансе «пыль» из-за погрешностей округления лотов.

    Метод возвращает List[Dict] — список параметров ордеров, которые нужно выставить.

    5. Сетевой слой: Изменения в CORE/executor.py
    Добавляем метод execute_tp_grid, который получает список ордеров от Фабрики и выставляет их на бирже.

    Используем батчинг или asyncio.gather с небольшим дилеем (как у тебя в легаси-коде было 0.15с, чтобы не словить Rate Limit от Phemex).

    При успешном ответе от Phemex (возврат orderID), экзекьютор под локом (self.tb._get_lock(pos_key)) записывает этот ID в pos.tp_orders.

    Если все ордера (или их критическая часть) выставлены, ставится флаг pos.tp_grid_initiated = True.

    6. Интеграция в Оркестратор (CORE/orchestrator.py)
    В главном цикле Game Loop или в пайплайне оценки (_evaluate_exit_scenarios):

    Проверяем: если pos.in_position == True и pos.tp_grid_initiated == False:

    Берем volume_24h из PriceCacheManager.

    Запрашиваем у GridTPFactory расчет сетки (получаем DTO).

    Закидываем этот список в _payloader или напрямую как асинхронную таску в OrderExecutor.

    Охота (HUNTING) полностью убирается. Состояние позиции становится пассивным (ждем исполнения лимиток биржей).

    7. Парсинг исполнения через WebSocket (CORE/models_fsm.py -> WsInterpreter)
    В методе _handle_order_update уже есть обработка ордеров.

    Если пришедший order_id есть в словаре pos.tp_orders и его статус FILLED:

    Делаем pos.tp_progress += 1.

    Пишем в лог: [BTCUSDT] Исполнен тейк-профит #1 по цене X.

    Если позиция полностью закрывается (объем стал 0 в _handle_position_update), она штатно удаляется из памяти оркестратором, а Аналитика (PerformanceTracker) записывает PnL.

    8. Дополнительно: Защита от рухнувших ордеров
